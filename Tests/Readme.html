Tests

Ce dossier contient tout ce qui concerne les tests: les données générées, les exécutables testeurs, les scripts, les résultats et les fichiers d’analyses réalisée après le test.

  • Donnees: les instances de problème générés, du scénario 1 au 8. Ces données seront normalement générées à chaque test par le testeur. Je les stocke ici pour besoins de tester des instances particulières.
  • TestCase1-6: Ce sont les tests réalisés pour la première partie de travail, sans Preprocessing. Ils ont été lancés sur mon portable au lieu de sur les machines OC. Normalement vous n’auriez pas besoin de regarder ces tests.
  • TestCase7_OC: A partir du TestCase7, les tests se font dans la salle OC. Ce test est le seul qui contient les résultat de H1, H2 et Cplex sans Preprocessing.
  • TestCases_Preprocessing: Les tests pour le Preprocessing. Les importants sont:
    • Test:5,13,15,17,18: Ce sont les tests pour Cut2 (1-cuts).
    • Test:9,10: Ces 2 tests sont faits pour chercher la fonction Seuil1 et Seuil2 pour Cut2.
    • Test 19: C’est un test du Seuil2, pour voir le pourcentage des variables fixées si on ne considère que les variables X.
    • Archive: Les autres tests moins intéressants.
    • Vectors: Ce sont les fichiers exportés qui contiennent une solution faisable trouvée par H2. Ils sont utilisés par AddMIPStart. Cf test 17, 18.
    • *.xlsx: Ce sont les fichiers d’analyse d’où viennent les tableaux dans le rapport. Mais c’est pas trop lisible car il y a trop de données à traiter…
  • Dlls: Pour lancer un TestCase, il faut rajouter les fichiers de ce dossier dans le dossier de test. Ce sont des dll requis. Je les extrais pour gagner de la place…

Dans le dossier de chaque TestCase, vous pouvez trouver normalement:

  • *.exe: le testeur généré à partir des programmes.
  • Run.bat: le script pour lancer le test, rediriger la sortie et éteindre la machine quand c’est fini.
  • Readme.txt: explique ce qu’on fait dans le test.
  • *.csv: le fichier de résultat.
  • *.xslx: le fichier de l’analyse, parfois intermédiaire.
  • test.log: fichier très utile. Il contient toutes les informations pendant l’exécution. Avec ceci, quand vous doutez sur le résultat d’un test, vous n’avez pas besoin de refaire le test pour savoir qu’est-ce qui s’est passé. Vous pensez peut-être générer ce log va gêner la mesure de temps dans le test? C’est pas un problème puisque chaque test dure un ou deux jours et Cplex va quand même afficher tout ça à l’écran si on ne fait pas cette redirection de sortie. Je pense qu’il faut continuer générer ce fichier pour garder la cohérence entre les résultats des tests…
